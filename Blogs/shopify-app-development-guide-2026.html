Shopify App Development Guide 2026: CLI, Remix & App Store Launch

Building a Shopify app in 2026 means starting from the Shopify CLI and Remix template, not building embedded admin UI from scratch. This guide walks through the real toolchain, the GraphQL basics you need, and the realistic cost and timeline for shipping a public app.

Setting Up: Shopify CLI and the Remix Template

Every Shopify app in 2026 should start from shopify app init, which scaffolds a working app using the official Remix template — Shopify's replacement for the older Node/Express boilerplate. The CLI handles the tedious parts automatically: registering your app with Partners, setting up OAuth, configuring webhooks, and spinning up a local tunnel so you can test inside a real development store without deploying anywhere first.

Remix was chosen as the default because it maps cleanly onto Shopify's embedded app model — server-side loaders fetch data via GraphQL before render, which avoids the client-side loading-spinner flicker that plagued older React-only embedded apps. If you're coming from a plain React or Next.js background, expect a short ramp-up period learning Remix's loader/action pattern before you're productive.

Admin & Storefront GraphQL API Basics

  • Admin GraphQL API is what your embedded app talks to for reading/writing store data — products, orders, customers, metafields. Shopify has fully deprecated new REST Admin API access for new apps, so GraphQL is not optional anymore; budget real learning time if your team is REST-native.
  • Storefront API is for customer-facing reads — product catalogs, cart operations — typically used by headless or custom storefront projects rather than admin-side apps. See our Shopify API guide for a full Admin vs. Storefront vs. REST breakdown.
  • Webhooks (now managed via GraphQL subscriptions rather than the old REST webhook endpoints) notify your app of store events — order creation, app uninstall, product update — and are mandatory for GDPR-mandatory compliance topics like customer data erasure requests.
  • API versioning follows a quarterly release cycle (e.g. 2025-10, 2026-01); pin your app to a specific version and plan a yearly migration pass, since Shopify sunsets older versions on a rolling basis.

The single hardest concept for developers new to Shopify's GraphQL API is cost-based rate limiting: instead of counting raw requests per minute, Shopify assigns each query a calculated "cost" based on the fields and nested connections you request, and throttles based on total cost consumed rather than request count. A query pulling ten nested product variants with images costs far more than a simple single-field lookup, even though both count as "one request." Design your queries to request only the fields you actually need, and implement retry-with-backoff logic for throttled responses rather than assuming every request will succeed immediately — production apps handling real merchant traffic hit this constantly.

Step-by-Step: From Idea to Installed App

1

Scaffold with the CLI

Run shopify app init, connect to a Partner account, and create a development store to test against — never build against a live production store.

2

Build the embedded UI with Polaris

Use Shopify's Polaris React component library and the App Bridge for navigation, so your app visually matches the native admin — this is a review requirement, not just a style preference.

3

Wire up GraphQL mutations/queries

Implement the actual feature logic against the Admin GraphQL API, handling rate limits (cost-based throttling, not simple request counts) gracefully with retry logic.

4

Add billing via the Billing API

Implement recurring or usage-based charges through Shopify's native Billing API rather than a third-party payment processor — this is required for any paid app distributed via the App Store.

5

Handle mandatory compliance webhooks

Implement the required GDPR webhooks (customer data request, customer redact, shop redact) — apps are rejected in review without these correctly implemented.

6

Submit for App Store review

Fill out the listing, provide test credentials and a demo video, and submit through Partners. Review checks functionality, performance, security, and design consistency against Shopify's requirements.

Building the Embedded UI: Why Polaris Matters

Polaris is Shopify's official design system — a React component library that mirrors the visual language of the Shopify admin itself (spacing, typography, colors, form controls). Building your app's screens from Polaris components rather than a custom design system isn't just a stylistic choice; it's effectively a review requirement, since Shopify explicitly checks that embedded apps feel native rather than like an iframe with a completely different design language bolted in.

Combined with App Bridge (which handles embedded navigation, modals, and the top bar inside the Shopify admin frame), Polaris also saves real development time — you're not rebuilding basic components like data tables, resource pickers, or banners from scratch. The trade-off is flexibility: highly custom or heavily branded UI needs deliberate workarounds, since Polaris is intentionally opinionated about layout and interaction patterns. Most successful apps use Polaris for 90%+ of their screens and reserve custom components only for the one or two features that genuinely need a different interaction model.

App Store Review: What Gets Apps Rejected

The most common rejection reasons aren't exotic bugs — they're avoidable process gaps: missing or broken GDPR webhooks, an uninstall flow that doesn't clean up properly, slow admin load times, or a UI that doesn't use Polaris/App Bridge consistently enough to feel native. Budget a full testing pass against Shopify's public app requirements checklist before your first submission — a rejected first review commonly adds one to two weeks to the timeline versus getting it right the first time.

  • Performance thresholds. Shopify measures embedded admin load time directly and will reject apps that load noticeably slower than the baseline — heavy client-side bundles or unoptimized GraphQL queries are the usual culprit.
  • Billing implementation errors. Apps that charge merchants outside the official Billing API, or that don't handle plan downgrades/cancellations correctly, are a fast rejection — this is treated as a trust and compliance issue, not just a bug.
  • Incomplete listing information. A vague app description, missing screenshots, or unclear pricing tiers on the listing page slow down review even when the app itself works fine — treat the listing content with the same care as the code.

Frequently Asked Questions

Do I need to know GraphQL to build a Shopify app in 2026?

Yes, effectively. Shopify has phased out REST Admin API access for new apps, so GraphQL fluency (queries, mutations, understanding cost-based rate limiting) is now a baseline requirement rather than optional.

Can I build a private app just for my own store without App Store review?

Yes — custom (unlisted) apps built for a single store via the Partner Dashboard skip public App Store review entirely. Review is only required for apps you intend to list publicly for other merchants to install.

How long does Shopify App Store review actually take?

Typically one to two weeks for a first submission if the app meets all technical and compliance requirements. Missing GDPR webhooks, broken uninstall flows, or inconsistent UI are the most common causes of an extended review cycle.

Should I use Remix or can I build a Shopify app with plain Node/Express?

You technically can build outside Remix, but you'll be reimplementing OAuth handling, session management, and webhook verification that the official template already solves. Unless you have a strong existing reason to use a different framework, starting from the Remix template saves weeks of foundational work and stays aligned with how Shopify documents and supports app development going forward.
